\chapter{Background}\label{ch:background}

This chapter provides the technical foundation for the approach in
\cref{ch:methods}: \sysml and its Interaction Model constructs, the
property graph data model, the static and dynamic analysis techniques
used, and how scenarios are scoped through graph queries. It is
background rather than contribution.

\section{SysML v2 and Interaction Models}\label{sec:sysml}

The \acl{OMG} \acl{SysML} version~2 (\sysml) is a general-purpose
modelling language for systems engineering, built on the Kernel Modeling
Language (\ac{KerML}) \cite{omg:sysmlv2}. Unlike its predecessor (SysML
v1, a profile of UML), \sysml defines its own textual and graphical
notation, enabling model interchange via a standardised textual syntax.

\subsection{Sequence Diagrams in SysML v2}

Interaction Models in \sysml are expressed as \emph{occurrence
specifications}: ordered sequences of message occurrences between
\emph{parts} (instances of blocks in the system structure). The
constructs used by the generator in this thesis (\cref{sec:sysml-impl})
are:
\begin{description}
  \item[\texttt{part def}~$:>$~\texttt{InteractionScenario}] A named
    container for a scenario-specific interaction model, analogous to a
    \ac{UML} interaction.
  \item[\texttt{Lifeline}] A participant in the interaction: an instance
    of a structural block along which message occurrences are ordered.
  \item[\texttt{SynchronousCall} / \texttt{ReturnMessage} /
    \texttt{AsynchronousMessage}] Message stereotypes, specialising a
    common \texttt{Message} definition, modelling a call, its return and
    an asynchronous send between two lifelines.
  \item[\texttt{from} / \texttt{to}] References binding a message to its
    sending and receiving lifelines, giving each occurrence its position
    in the ordered sequence.
\end{description}

An example in \sysml textual notation, corresponding to scenario S1
(Aggregator notification chain) and following the same stereotype-based
form the generator emits (\cref{sec:sysml-impl}):

\begin{minted}[fontsize=\footnotesize]{text}
package AggregatorNotification {
    part def SynchronousCall :> Message { ... }

    part def S1_AggregatorChain :> InteractionScenario {
        part runner : Lifeline;
        part agg    : Lifeline;
        part log    : Lifeline;

        part m1 : SynchronousCall {
            ref from : Lifeline = runner;
            ref to   : Lifeline = agg;
            attribute :>> label = "1. getAll";
        }
        part m2 : SynchronousCall {
            ref from : Lifeline = runner;
            ref to   : Lifeline = log;
            attribute :>> label = "2. stream";
        }
        part m3 : SynchronousCall {
            ref from : Lifeline = runner;
            ref to   : Lifeline = log;
            attribute :>> label = "3. flush";
        }
        part m4 : SynchronousCall {
            ref from : Lifeline = runner;
            ref to   : Lifeline = log;
            attribute :>> label = "4. instance";
        }
    }
}
\end{minted}

\section{Property Graphs}\label{sec:property-graphs}

A \acl{PG} is a directed, attributed multigraph in which both nodes and
edges carry arbitrary key-value properties \cite{angles:pg-survey}.
Formally, a \ac{PG} is $G = (V, E, \lambda, \mu)$, where $V$ is the set
of nodes; $\lambda(v) \in \mathcal{L}$ assigns a label from the label set
$\mathcal{L}$ to each node $v \in V$; $E \subseteq V \times \mathcal{R}
\times V$ is the set of directed edges, each carrying a relation type $r$
from the relation-type set $\mathcal{R}$; and $\mu : (V \cup E) \times
\mathcal{K} \rightharpoonup \mathcal{V}$ is a partial function assigning
property values from value set $\mathcal{V}$ to nodes or edges, indexed
by property keys from key set $\mathcal{K}$.

\neo is the most widely deployed \ac{GDB} implementing the \ac{PG} model,
queried via the Cypher declarative query language \cite{neo4j:cypher}. A
\ac{PG} was chosen over a relational or document store because it lets
heterogeneous analysis artefacts (functions, files, modules and trace
events) be traversed efficiently along typed relationships. In this
thesis, \neo is the store the Renaissance AST matcher populates
(\cref{sec:static-analysis}), and scenario scoping is itself expressed as
a Cypher query over that store (\cref{sec:scenario-scoping}).

\section{Static Call-Graph Analysis}\label{sec:static-analysis}

A \emph{call graph} is a directed graph in which nodes represent
functions (or methods) and edges represent potential call relationships
\cite{grove:callgraph}: an edge $(f, g)$ exists if and only if caller $f$
may call callee $g$ under some execution.

\subsection{Clang and the Renaissance AST Matcher}

Clang, the \Cpp\ frontend of LLVM, is a parser: it builds the full
\ac{AST} for a translation unit and exposes an \ac{AST} traversal
\ac{API} \cite{llvm:clang}. \emph{Renaissance} is a semantic-level
\Cpp\ analysis methodology already in use at TNO-ESI, which resolves
call, containment and inheritance relationships by matching concrete
syntax patterns over the AST rather than by walking raw parse-tree nodes
directly. This thesis re-implements the Renaissance AST matcher on top
of Clang, using Clang's AST matcher API to perform the same
concrete-syntax-pattern matching and populate the property graph
(\cref{sec:pg-decision}). The resulting extractor resolves the full
\ac{AST}, including template instantiations, overload candidates and
macro expansions, and emits the schema described in \cref{sec:pg-schema}.

\subsection{Limitations in Event-Driven Systems}

Event-driven systems that route inter-component communication through
callback registration or publish-subscribe indirection pose a specific
problem for static call-graph construction. In CPSCore, subscriptions
are registered via the IDC layer:
\begin{minted}{cpp}
IDC::subscribeOnPacket(channelId, handlerSlot);
\end{minted}
Standard \ac{AST} traversal identifies the call to
\texttt{subscribeOnPacket} but cannot resolve which component receives a
packet published on the same channel, since publisher and subscriber are
tied together only by a shared runtime identifier, not a call expression.
The resulting static call graph contains edges \emph{into} the IDC/Redis
layer but lacks the cross-component routing information interaction
model reconstruction needs. \Cref{sec:macro-resolution} addresses this
by matching publisher and subscriber call sites on that shared
identifier, the general mechanism behind CPSCore's own IDC/Redis
implementation (\cref{sec:idc,sec:pubsub-background}).

\section{Execution Trace Analysis}\label{sec:trace-analysis}

An \emph{execution trace} is a temporally ordered sequence of events
recorded at runtime, typically by instrumenting the system under
analysis with logging or profiling statements. In
this thesis, traces take the form of timestamped log entries emitted at
instrumented call sites rather than at function entry/exit
(\cref{sec:csp-matcher}), so a trace records that a specific
cross-component call executed, not merely that a function was entered;
each entry carries the caller and callee component identity and the
name of the interaction invoked.

\subsection{Challenges with Incomplete Traces}

Practical execution traces are almost always incomplete: not every
component is instrumented, recording overhead can drop log entries, and
a single scenario recording covers only a subset of the system's
behaviour. These gaps are what motivate the hybrid approach: the static
call graph is hypothesised to supply structural completeness where the
trace is thin, and the trace to supply scenario specificity where the
static graph over-approximates. \Cref{sec:h1} tests this hypothesis
directly; the result is a qualified confirmation rather than a clean
division of labour, since static analysis can itself miss interactions
resolved only through indirection (\cref{sec:h1-c4c3}), so the two
sources are complementary in aggregate rather than each unconditionally
reliable in its own dimension.

The hybrid approach exploits this complementarity in both directions.
Static-analysis-guided instrumentation uses the architectural interface
map (\cref{sec:csp-matcher}) to decide where to place trace probes,
improving trace completeness relative to unguided instrumentation
(H2, \cref{ch:experiments,ch:analysis}). Conversely, static-analysis
pruning uses the scenario-scoped query (\cref{sec:scenario-scoping}) to
reduce the over-approximate call graph down to the interactions relevant
to the scenario at hand, rather than the full codebase.

\section{Scenario Scoping through Graph Queries}\label{sec:scenario-scoping}

Reconstructing a diagram for a single scenario requires narrowing a
large call graph down to the interactions that belong to it. Rather than
a formal program-slicing procedure \cite{weiser:slicing} over a program
dependence graph \cite{ferrante:pdg}, this thesis performs scenario
scoping directly as a declarative graph query:
a scenario's scope is expressed as a set of participating components
(\eg \texttt{Synchronization} calling into \texttt{Aggregation} and
\texttt{Logging}), and a scenario-scoped Cypher query retains only the
call edges whose caller and callee belong to those components, keeping
what is relevant to the scenario and discarding interactions belonging
to other, concurrently exercised behaviour.

The concrete query is described in \cref{sec:combine-scope}. It operates
over the static property graph rather than a source-level program
representation, and the same component-scope constraint is applied to
the trace-derived sequence table.

\section{Publish-Subscribe via Callback Registration}\label{sec:pubsub-background}

Many event-driven \Cpp\ systems decouple component communication through
a publish-subscribe pattern, where subscriptions are registered at
runtime via callback slots rather than direct function calls, making the
topology implicit in the source and unrecoverable from call sites alone
(\cref{sec:static-analysis}). CPSCore implements this through its
\ac{IDC} layer, which delegates to a pluggable \texttt{INetworkLayer}
backed by \texttt{boost::signals2} and Redis channels
(\cref{sec:idc}). Recovering the subscription topology means matching
publisher and subscriber call sites by their shared channel identifier,
the approach taken in \cref{sec:macro-resolution}.
