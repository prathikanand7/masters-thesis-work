\chapter{Conclusion}\label{ch:conclusion}

\section{Summary \& Findings}

This thesis addressed the problem of automatically reconstructing
scenario-specific Interaction Models from \Cpp\ codebases and execution
traces in event-driven systems.
The core challenge is that static analysis alone produces over-approximate,
scenario-agnostic graphs, while trace-based analysis alone produces
under-complete models due to limited runtime instrumentation.
No prior approach simultaneously addresses publish-subscribe topology
recovery and scenario filtering via graph slicing, nor targets \sysml
as the output formalism.

The proposed hybrid approach constructs a unified \ac{PG} in \neo by
integrating a static call graph (including IDC pub-sub topology-resolved
edges) with timestamped execution trace events.
Dependence-based slicing is then applied to the fused graph to isolate the
scenario-relevant subset.
The sliced graph is projected into \sysml textual notation.

The approach was developed and evaluated on \emph{CPSCore}, an open-source
event-driven \Cpp\ framework, across four scenarios: three
happy-path scenarios occurring naturally in CPSCore's existing code,
plus one (S4) purpose-built to isolate a fusion precondition the other
three cannot exercise.
Three hypotheses were tested to answer the research questions:

\begin{description}
  \item[H1 (confirmed).]
    Fusion produces more accurate interaction models than any single
    evidence source.
    The full pipeline (C5) achieves the highest macro-F1 of any condition
    tested (0.964) with perfect recall on every scenario.
    On the three happy-path scenarios, the static graph is already
    reachability-complete, so fusion is redundant with static analysis
    alone (C4$=$C2). On the constructed scenario~S4, where a
    publish-subscribe boundary and log suppression each hide a different
    interaction, fusion is strictly additive: F1(C4)$=$1.000, versus
    0.667 for either single source.
    This confirms that fusion's value activates precisely when no single
    source can see the full picture, and is otherwise safely redundant
    (\cref{sec:h1-results}).

  \item[H2 (confirmed).]
    Code-graph-guided instrumentation eliminates all measured noise
    (50\% fewer retained nodes, 70\% fewer edges) without losing coverage.
    A blinded six-rater expert panel confirmed perceptual superiority:
    guided output was rated strictly higher in 47 of 54 comparisons, with
    one reversal traceable to a specific design trade-off (collapsing
    repeated property-read calls into a single edge).
    The quantitative and perceptual results align: noise removal improves
    usability (\cref{sec:h2-results,sec:h2-qual}).

  \item[H3 (confirmed).]
    In a holistic five-condition experiment, the reconstructed diagram
    alone achieves perfect component recall, interaction recall, and rated
    completeness on every scenario, matching or exceeding all combinations
    that include source code and traces.
    Conditions without the diagram consistently miss a suppressed-but-source-visible
    interaction that the diagram states explicitly.
    The diagram is therefore the distinguishing factor, not merely a
    supplement (\cref{sec:h3-holistic}).
\end{description}

As an exploratory complement to H1--H3, a practitioner hands-on session and
interview study with four engineers on a second, much larger industrial
codebase (\cref{sec:practitioner}) provides encouraging but convergent
evidence that the same pattern holds at industrial scale.
The key scenario exercise involved a proxy-leak defect whose root cause is an
implicit event-driven interaction invisible without reconstructing the
call chain. It was resolved using only the static graph. This mirrors H1's
finding that static analysis preserves fault-path recall where traces
cannot.
Furthermore, the industrial system's architecture (proxy-based, virtual
interfaces, dynamic service binding) is precisely the kind of system
where H1 predicts C4$>$C2 would materialise
(\cref{sec:cpscore-scale,sec:h1-c4c2}).

A single cross-cutting pattern unifies all four evidence sources
(\cref{sec:discussion}): \textbf{additional evidence helps exactly when
the primary evidence source is missing something, and is otherwise
redundant.}
Fusion helps on fault paths but not happy paths (H1); diagrams help
comprehension when traces are incomplete but not when they are already
sufficient (H3); guided instrumentation improves usability by removing
what is irrelevant (H2); and practitioners solved a fault-path defect
using the exact evidence source (static graph) that H1 identifies as
the recall-preserving complement to incomplete traces.
This pattern also generates a concrete, falsifiable prediction for
future work: on systems with cross-module virtual dispatch, fusion
should show C4$>$C2 on \emph{all} scenario types, not just fault
paths (\cref{sec:future-work}).

\section{Practical Implications}

The pipeline enables an engineer working on an event-driven \Cpp\ system to
obtain a scenario-specific \sysml\ interaction diagram by issuing
natural-language queries against the property graph, rather than spending
several working days manually tracing call chains, cross-referencing header
files, and correlating log output.
On CPSCore, the static extraction skill populated 2,574 ordered interactions
across 121 components in a single Cypher query execution. The instrumentation
skill required editing only nine function bodies across five files, guided
entirely by a machine-generated mapping CSV.
Because the pipeline is codebase-agnostic (skills query the live schema, not
hardcoded labels), switching to a new target requires only re-running the
static extractor and re-invoking the same skills.
The practitioner session confirmed this transfer cost is low in practice:
four engineers on a much larger industrial codebase used the tool
productively within a single session, and the structural and dependency
diagrams were rated consistently useful for architectural comprehension
and defect localisation.

\section{Contributions}

The primary contributions of this thesis are organised into three research
contributions and three engineering artefacts:

\paragraph{Research contributions.}
\begin{enumerate}
  \item \textbf{A provenance-preserving property graph schema} that
    co-locates static call-graph and runtime trace artefacts in a single
    \neo graph, tagging each edge by its evidential source (static,
    dynamic, or both). This schema enables both fusion and slicing to
    operate on a single data structure without losing traceability
    (\cref{sec:pg-schema}).

  \item \textbf{A scope-based graph slicing algorithm} adapted to
    heterogeneous property graphs mixing static and dynamic edges,
    enabling scenario-specific filtering without re-running the analysis
    from scratch (\cref{sec:fusion-slicing}).

  \item \textbf{An empirical characterisation} of when hybrid synthesis
    helps and when it does not: fusion is protective on fault paths and
    additive under cross-source indirection boundaries (H1); guided
    instrumentation eliminates noise without coverage loss (H2); and
    reconstructed diagrams improve downstream comprehension specifically
    when runtime evidence is suppressed (H3).
\end{enumerate}

\paragraph{Engineering artefacts.}
\begin{enumerate}\setcounter{enumi}{3}
  \item A \textbf{pattern-based AST instrumentation tool}
    (\texttt{csp\_matcher}) enabling reproducible and reversible trace
    probe injection via a declarative find/replace language
    (\cref{sec:csp-matcher}).

  \item A \textbf{composable agent skill architecture} (twelve skills)
    orchestrating the full pipeline from source to \sysml\ via
    natural-language queries, requiring no code changes when switching
    target codebases (\cref{sec:skill-design}).

  \item A \textbf{publish-subscribe topology recovery strategy} resolving
    implicit callback routing by matching subscription and publication
    call sites by channel identifier (\cref{sec:macro-resolution}).
\end{enumerate}

\section{Limitations \& Threats to Validity}

The most significant limitation is scale: all three hypotheses were
evaluated on a single open-source system (CPSCore, $\approx$13\,kLOC)
across four scenarios.
CPSCore's absence of cross-module virtual dispatch means the static call
graph is already reachability-complete, explaining why fusion is
redundant on happy-path scenarios. In larger systems with virtual dispatch
or dynamic plugin loading, the fusion benefit would likely be more
pronounced.
The practitioner session on an industrial-scale codebase
(\cref{sec:practitioner}) provides encouraging qualitative evidence of
transfer, but does not substitute for a controlled replication.

The H3 experiment used the same LLM as both generator and evaluator,
introducing a potential self-consistency bias.
The H2 six-rater panel preserved effect direction on every dimension,
but a larger sample would strengthen confidence in magnitude.
Ground-truth diagrams are inherently subjective, and the evaluation
metric does not capture temporal ordering accuracy.

A complete discussion of threats to validity, including researcher degrees
of freedom identified and corrected during H1/H2, is given in
\cref{ch:analysis}.

\section{Future Work}\label{sec:future-work}

Several directions are open for future investigation:

\begin{description}
  \item[Controlled evaluation on a second system.]
    The practitioner session (\cref{sec:practitioner}) suggests the
    approach transfers to larger, industrial codebases, but only
    qualitatively. Extending the fault-path/dual-gap scenario set beyond
    the single constructed scenario (S4) currently reported would be a
    natural first step. Applying the full, pre-registered
    H1/H2/H3 hypothesis tests to that codebase, or to other event-driven
    \Cpp\ systems (for example, using different publish-subscribe frameworks such as
    ROS~2 or MQTT), would turn this encouraging signal into a controlled
    generalisability result.

  \item[Diagram labelling and interactivity.]
    The practitioner interviews (\cref{sec:interview-findings})
    independently converged on two concrete requests: (i) richer, less
    abstract event/message labels on sequence diagrams (\eg inline
    source class and line number on hover); and (ii) interactive
    exploration (search, filter, collapse/expand) rather than static,
    one-shot diagrams. Both are natural extensions of provenance data the
    pipeline already carries (\cref{sec:utility}).

  \item[Temporal ordering accuracy.]
    The current evaluation abstracts over message ordering.
    Developing a metric that accounts for the relative order of interactions
    in the generated diagram would provide a more complete picture.

  \item[Incremental updates.]
    As the codebase evolves, the \ac{PG} should be updatable incrementally
    rather than re-built from scratch.
    Neo4j's support for live graph updates makes this tractable, but the
    slicing invalidation logic would need to be designed carefully.

  \item[LLM-assisted diagram naming.]
    Function names in the generated diagrams are currently taken verbatim
    from the source code.
    A post-processing step using a \ac{LLM} could produce more
    human-readable message labels, improving the practical utility of the
    output.

  \item[Integration with Renaissance DB.]
    The \ac{GDI} project maintains an architecture model database
    (Renaissance DB) that could serve as a richer source of module-level
    metadata, potentially reducing the number of unresolved black-box
    components.
\end{description}
