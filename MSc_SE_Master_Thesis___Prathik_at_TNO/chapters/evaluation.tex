\chapter{Evaluation}\label{ch:analysis}

This chapter evaluates the \ac{GDI} pipeline against the three hypotheses
(H1, H2, H3) that operationalise the research questions from
\cref{ch:introduction}. It reports the design and results of each
experiment on the CPSCore case study, a complementary practitioner
session on a second industrial codebase and a discussion of threats to
validity, including several methodological risks caught during the
experiments themselves.

\section{Evaluation Design}\label{sec:eval-design}

\begin{itemize}
  \item \textbf{H1} (\cref{sec:h1}) compares reconstruction accuracy
    across conditions C1--C5 (\cref{sec:conditions}) on four scenarios
    (\cref{sec:scenarios}): three occurring naturally in CPSCore (S1--S3),
    plus one purpose-built scenario (S4) exercising a precondition S1--S3
    cannot.
  \item \textbf{H2} (\cref{sec:h2}) compares guided against full
    instrumentation on structural compression, noise and coverage, with a
    blinded expert rating of the resulting diagrams.
  \item \textbf{H3} (\cref{sec:h3}) tests whether the reconstructed models
    improve an agent's performance on a scenario-comprehension task,
    relative to reasoning over raw source code.
\end{itemize}

\Cref{sec:rq-answers} draws the two research questions' answers together
from these experiments.
\Cref{sec:practitioner} reports a fourth, exploratory data point ---
practitioner feedback from a second, much larger industrial codebase ---
kept separate from H1--H3 since it was not run as a controlled or
pre-registered comparison.

\section{H1: Does Combining Static and Dynamic Evidence Improve Reconstruction Accuracy?}\label{sec:h1}

\textbf{Hypothesis.} Combining static analysis and dynamic instrumentation
(C4) improves reconstruction accuracy over any single source (C1--C3)
alone.

\subsection{Metric}

For every condition and scenario, the generated \sysml sequence diagram
is compared against a manually authored reference diagram over
\emph{interaction edges} (unique ordered $\langle\text{sender},
\text{receiver}\rangle$ pairs):

\[
  P = \frac{|\text{gen} \cap \text{ref}|}{|\text{gen}|}, \quad
  R = \frac{|\text{gen} \cap \text{ref}|}{|\text{ref}|}, \quad
  F_1 = \frac{2PR}{P + R}
\]

Reference diagrams were authored before any condition was scored against
them, to avoid tuning the reference to a particular output.
This formulation follows the standard information-retrieval
convention~\cite{manning:ir}, mirrors its recent use for comparing
static against dynamically-sampled call graphs~\cite{helm:totalrecall},
and matches the convention used within the sequence-diagram
reconstruction literature itself~\cite{taniguchi:seqtrace}.

\subsection{Methodological Corrections}\label{sec:h1-corrections}

Four risks were identified and corrected while H1 was being run:

\begin{description}
  \item[Ground-truth circularity.] An early reference set was partly
    derived from the same static extraction used to produce C2/C4/C5,
    which would have inflated their scores by construction. The
    references were reauthored independently and verified against
    Doxygen-generated call graphs, an unrelated static analysis tool.
  \item[\texttt{SCENARIO\_PRIMARY} scope boundary.] The in-scope
    boundary was initially defined \emph{after} inspecting a preferred
    condition's results, risking a favourable definition. It is now
    fixed from the scenario definition alone, before any condition runs.
  \item[F1 aggregation.] Macro-F1 (mean of per-scenario F1) was adopted
    over micro-F1 to avoid large scenarios dominating the
    aggregate~\cite{sokolova:metrics}; the two agree to within
    $\pm 0.013$.
  \item[Constructed-scenario disclosure.] S4 was added after S1--S3
    were scored, to test whether combining could exceed \emph{both}
    single sources at once --- a property no naturally-occurring CPSCore
    path exhibits. It is reported as its own stratum
    (\cref{tab:h1-results}) rather than blended into the S1--S3 aggregate.
\end{description}

\subsection{Discussion: Why C4 $=$ C2 on S1--S3}\label{sec:h1-c4c2}

On the three scenarios found naturally in CPSCore's codebase, C2 and C4
are numerically identical (0.867/1.000/0.917). This follows directly
from a structural property of CPSCore itself (\cref{sec:cpscore-scale}):
the codebase exhibits no cross-module virtual dispatch, so every call
target is statically determinate, and every edge the trace observes was
already present in the static graph:

\[
  \text{C3} \subseteq \text{C2} \quad \text{(on S1--S3)}
  \;\Longrightarrow\;
  \text{C4} = \text{C2} \cup \text{C3} = \text{C2} \quad \text{(on S1--S3)}
\]

A system with cross-module virtual dispatch, dynamic plugin loading, or
reflection-based routing could plausibly break this containment, since
traces there might observe targets static analysis cannot resolve.
CPSCore's found-in-the-wild code does not have this property on S1--S3,
so C4$=$C2 there is exactly what the underlying set relationship
predicts. \Cref{sec:h1-c4c3} constructs a scenario (S4) with this
property deliberately.

C3 alone is the best-performing condition on these scenarios
(macro-F1 1.000, against 0.917 for C2/C4), not merely a redundant subset
of the static graph. The gap traces to two static over-approximation
false positives on S3 (\texttt{flush}, \texttt{instance}) that C2/C4
carry but C3 never sees, since those calls do not execute during the S3
test. Combining is therefore not protective on S1--S3 --- there is
nothing suppressed to protect against --- and is mildly harmful to
precision relative to the trace alone. The stronger case for combining
rests on S4 (\cref{sec:h1-c4c3}).

The same S3 false positives explain C2/C4's precision deficit
(0.867, not 1.000): both calls are genuine source-level call sites, but
belong to \texttt{SynchronizedRunnerMaster.runStage} rather than S3's
primary function \texttt{SimpleRunner.runStage}. This reflects a more
general limitation --- static analysis has no way to scope itself to a
single scenario --- which is the over-approximation problem motivating
C5.

\subsection{Discussion: S4 and the Limits of C4 $=$ C2}\label{sec:h1-c4c3}

Since S1--S3 all satisfy $\text{C3} \subseteq \text{C2}$, combining is
bounded above by C2 there. S4 (\cref{sec:scenarios}) removes that ceiling
on both sides at once, via a \texttt{boost::signals2} publish/subscribe
bridge between \texttt{StageEventBridge} (Synchronization) and
\texttt{StageEventListener} (Aggregation), where each single-source
condition is blind to a different edge:

\begin{itemize}
  \item C2 sees \texttt{stream} (an ordinary, statically-resolvable log
    call) but not \texttt{onStageEvent}, whose dispatch is only
    established via \texttt{connect()} at runtime: P/R/F1~$=$~1.000/0.500/0.667.
  \item C3 sees \texttt{onStageEvent} but not \texttt{stream}, suppressed
    by a \texttt{LogLevel::NONE} idiom that makes the
    \texttt{CPSLOG\_ERROR} call a no-op: P/R/F1~$=$~1.000/0.500/0.667.
  \item C4, the union, recovers both: P/R/F1~$=$~1.000/1.000/\textbf{1.000}.
\end{itemize}

F1(C4)~$=$~1.000 strictly exceeds both F1(C2) and F1(C3) on S4,
consistent with the hypothesis S4 was built to test: when static and
dynamic analysis each miss a genuinely different edge, combining adds
something new rather than merely guarding against suppressed evidence.
All five S4 conditions were run for real, not simulated.

C5 inherits this result: given C2/C3 as separately labelled evidence
(\cref{sec:h1-c5}), it retains both edges on S4 and reaches F1~$=$~1.000,
matching C4.

\subsection{Discussion: C5 Source-Aware Synthesis}\label{sec:h1-c5}

C5 gives the agent C2 and C3 as separately labelled evidence, each edge
tagged with whether the other source also observed it, and asks the
agent to reason about \emph{why} a single-source edge might be missing:
genuine runtime suppression, indirect dispatch, or static
over-approximation. A single-source edge is not treated as automatically
weak evidence. The response also includes an ordered call sequence,
using the real trace order as backbone with retained static-only edges
inserted at the position the scenario description implies.

C5 matches C4's recall (R~$=$~1.000) on all four scenarios, retaining
\texttt{CPSLogger::instance} on S1 and \texttt{stream} on S4. The one
imprecision is on S3, where the heuristic that correctly recovers S4's
suppressed edge is misapplied to \texttt{flush}: a real edge, but from a
different function, outside S3's scope. C2's element list carries
source, target and interaction name but not the calling function, so
``suppressed-but-in-scope'' and ``real-but-out-of-scope'' look identical
to this reasoning step. Across S1--S4, C5 reaches macro-P/R/F1 of
0.938/1.000/\textbf{0.964}, the best F1 of any condition, ahead of C4's
raw union (0.938) and C3's raw trace (0.917, held back by its S4 miss).

In short: combining is redundant on S1--S3 and strictly additive only on
the constructed S4 case; source-aware synthesis preserves that recall
while partially correcting C4's S3 precision loss, at the cost of one
disclosed imprecision that a component-level (rather than function-level)
source label cannot resolve.

\section{H2: Does Guided Instrumentation Improve Scenario Focus?}\label{sec:h2}

\textbf{Hypothesis.} Code-graph-guided instrumentation
(\cref{sec:guided-vs-full}) suppresses irrelevant execution detail
relative to full, unguided instrumentation.

H2 has a quantitative arm (structural compression, noise, coverage) and
a qualitative arm (blinded expert rating). Both were needed: H2 is
fundamentally a claim about human-perceived usability, which the
quantitative metrics alone cannot establish.

\subsection{Metrics}

Let $G_{\text{full}}$, $G_{\text{guided}}$ denote the property-graph
subsets from full and guided instrumentation, and $R$ the
scenario-relevant reference node set (shared with H1):

\[
  \text{NodeReduction} = 1 - \frac{|V(G_{\text{guided}})|}{|V(G_{\text{full}})|},
  \qquad
  \text{EdgeReduction} = 1 - \frac{|E(G_{\text{guided}})|}{|E(G_{\text{full}})|}
\]

\[
  \text{Noise}(G) = \frac{|V(G) \setminus R|}{|V(G)|},
  \qquad
  \text{Coverage}(G) = \frac{|V(G) \cap R|}{|R|}
\]

Noise and Coverage are the retained-set analogues of $1-\text{Precision}$
and Recall~\cite{manning:ir}; the noise concept is adapted from its use
in architecture recovery evaluation~\cite{constantinou:noise}, applied
here to instrumentation scoping rather than clustering. A configuration
that reduces noise by discarding relevant nodes along with irrelevant
ones is not an improvement --- coverage rules that failure mode out.

\subsection{Circularity Check}\label{sec:h2-corrections}

Guided instrumentation restricts probes to the scenario's static scope,
and the reference set $R$ used to score noise/coverage is derived from
the same scope boundary. A noise score of exactly $0$ for
$G_{\text{guided}}$ is therefore partly guaranteed by construction. This
is addressed in two ways: the result is reported explicitly as
\emph{partly by construction} rather than a clean finding; and scenario
S3, whose scope and reference were not defined from the same pass,
serves as a genuine falsification test. S3 reproduced the same pattern,
which is what allows the result to be reported as more than circular.

excess events for guided instrumentation to remove.

\subsection{S4: A Discovered Scoring Artifact}\label{sec:h2-s4}

Unlike S1--S3, whose full trace is a direct copy of a pre-existing
capture, S4's production code did not exist when that capture was
taken; its full-instrumentation trace was constructed as the
pre-existing capture plus S4's own test-captured row --- the best
available simulation of a full instrumentation pass over the larger
test suite.

Scoring this file gave an unexpected result: S4-full shows
coverage~$=$~1.000, appearing to recover \texttt{stream} as well as
\texttt{onStageEvent}. This is a false positive. The pre-existing
capture already contains several unrelated
\texttt{Synchronization$\to$Logging:stream} rows from other functions
(\eg \texttt{SimpleRunner.runStage}) under normal logging conditions ---
none originating from \texttt{StageEventBridge::publishStage}, the
function that actually matters for S4. The scoring convention matches
only on the bare (source, target, interaction-name) triple, deliberately,
to mirror how a human would describe an interaction
(\cref{sec:h1-corrections}); under this convention, any unrelated row
satisfies the reference requirement.

The genuine S4 fault-path call remains absent from every trace, full or
guided, suppressed by the same runtime gate. The guided trace cannot
exhibit this artifact by construction, since it only ever contains rows
captured during the S4 test itself; it correctly reports
coverage~$=$~0.500. This strengthens rather than weakens the H2 argument:
it surfaces a failure mode of full instrumentation beyond noise ---
pooling events across an unscoped test suite risks an unrelated call
coincidentally satisfying a reference triple by bare name, producing
unverifiable, inflated coverage rather than a true positive. S4-full's
raw coverage is retained in \cref{tab:h2-per-scenario} for transparency,
but neither strategy actually recovers \texttt{stream} for S4, so S4 is
excluded from the S1--S3 aggregate in \cref{tab:h2-results}.

\subsection{Qualitative Arm: Blinded Expert Rating}\label{sec:h2-qual}

Structural compression alone cannot establish that guided instrumentation
produces a \emph{more usable} diagram --- that is a claim about human
perception. A blinded A/B protocol tested this directly: six external
domain engineers, with no prior knowledge of the codebase, independently
rated diagrams from both configurations without being told which was
which, using a rating form and a private decode key (held separately) to
keep the assignment blind, with the A/B label randomised per scenario.
All six returned completed forms scoring S1--S3 on accuracy, readability
and practical usefulness, plus free-text comments and an overall
preference.

where the reduction discards a property-read count several raters found
independently useful for debugging.

\section{H3: Do Reconstructed Models Improve Task Performance?}\label{sec:h3}

\textbf{Hypothesis.} Agents equipped with reconstructed structural and
sequence diagrams perform scenario-comprehension tasks more accurately
than agents relying only on source code and traces.

H3 was scoped to a single downstream task (Scenario Comprehension) to
keep the evaluation tractable. A single holistic experiment isolated
source code, execution traces, and the C5 diagram individually and in
combination, across S1--S4.

\subsection{Protocol}

Ground truth was derived from the H1 reference diagrams (the same
\texttt{elements.json} files, manually source-read) converted into
expected components, expected interactions and a reference explanation.
For each scenario, an agent was prompted to identify components,
enumerate interactions and explain the execution flow under one of five
conditions:

\begin{description}
  \item[Source only.] Source snippets only.
  \item[Trace only.] Runtime trace subset only.
  \item[Diagram only.] C5 interaction list only.
  \item[Source + Trace.] Both.
  \item[Source + Trace + Diagram.] All three.
\end{description}

Three conditions (trace only, source+trace, source+trace+diagram) reuse
real agent responses from earlier isolated comparisons on this same
harness; source only and diagram only were newly collected for this
design, so every point in the design space is populated with real,
non-fabricated data from the same model and prompt family.

Component and interaction recall apply the same formulation as H1.
Completeness is scored 1--5. The three reused conditions were rated
blind (responses anonymised, decode key held separately), following
standard blinding practice~\cite{wohlin:experimentation}; the two new
conditions were rated in a single non-blind pass by the same rater and
rubric, disclosed as a limitation (\cref{sec:threats-rdof}).

Diagram only and source+trace+diagram are identical on every measure:
with no source and no trace access, the C5 list alone yields perfect
component recall, interaction recall and completeness on every scenario,
including S4. Adding source and traces on top neither helps nor hurts.
This shows the reconstructed diagram is not merely a helpful
supplement: it can be a \emph{sufficient} basis for comprehension.

Source only has the weakest interaction recall (0.800): the agent
under-reports the log-call interaction (\texttt{RAIILogStream.stream})
on S1 and S4 despite explicit source comments identifying it. This does
not appear to be a lack of access so much as a tendency for
source-grounded narration to foreground the trace-confirmed,
listener-connected path and omit secondary, log-only branches. Trace
only avoids this on S1 but repeats it on S4, where the trace never
captures the suppressed edge. Diagram only avoids it everywhere, since
C5 surfaces the interaction as an explicit labelled edge regardless of
whether it was exercised at runtime.

Source+trace scores identically to trace only (0.917/0.900): inspecting
the response suggests the agent treats the trace as the more
authoritative source whenever the two disagree, rather than reconciling
them as independent evidence. Despite an explicit source comment marking
the suppressed call, the response narrates only the trace-confirmed
path. Adding the diagram removes this asymmetry rather than
out-arguing it: C5 presents every edge as a flat, equally weighted fact
with no ``this is what actually happened'' framing to anchor on, leaving
nothing for the agent to rank above the suppressed edge.

Completeness mirrors the recall pattern: the two diagram-inclusive
conditions score a perfect 5.00 on every scenario, while the three
without the diagram are tied at 4.25, each losing a point specifically
on S4 for the same reason.

\subsection{H3 Conclusion}

H3 is supported. Across all four scenarios, diagram only and
source+trace+diagram achieve identical, perfect scores on every metric,
matching or outperforming every condition that additionally includes
source and/or traces. The three conditions without the diagram are held
to 0.917/0.800--0.900 recall by the same recurring failure ---
under-reporting the interaction the diagram states explicitly.

This mirrors the H1 finding (\cref{sec:h1-c5}): source-aware synthesis
(C5) is what lets the reconstructed model retain the evidence that makes
it useful downstream. The holistic result functions as an independent
cross-validation, via a second measurement instrument (a comprehension
task rather than an edge-set F1 score), suggesting the same mechanism
behind H1's reconstruction accuracy also improves a consuming agent's
task performance. The benefit appears to be one of \emph{presentation}
rather than \emph{access}: source-only and source+trace conditions have
the same information available and still fail to surface it, while the
diagram's flat, unranked facts counteract the tendency to narrate around
the trace-confirmed path. Limitations (one constructed scenario, two
non-blind conditions) are discussed in \cref{sec:threats-rdof}.

\section{Practitioner Validation: A Second Industrial Codebase}\label{sec:practitioner}

H1--H3 validate the approach in depth on one system (CPSCore) under
controlled conditions. This section reports a complementary, exploratory
data point. \emph{Renaissance MCP}, the tool built in this project,
exposes the Renaissance static-extraction backend and graph-querying
approach as an MCP integration inside GitHub Copilot, and was trialled
by practitioners at an industrial partner on a second \Cpp\ codebase
several orders of magnitude larger than CPSCore.

This is not a repeat of H1/H2/H3: it is an unblinded, semi-structured
hands-on session and interview study with four engineers, exercising
only the static-extraction and graph-querying capability. No runtime
instrumentation or dynamic analysis was involved, so it speaks to the
static half of the approach only, and is reported because it bears
directly on the external-validity gap discussed later
(\cref{sec:threats-rdof}): whether the static-analysis capability holds
up on a second, much larger system. Codebase and defect details are kept
generic to avoid disclosing partner-internal specifics.

\subsection{Session Design}

Each session had four parts: tool installation, an \emph{estimation
game} (guessing then checking quantitative facts about the codebase), a
\emph{scenario exercise} investigating a real, previously-resolved
engineering issue, and a wrap-up discussion on the tool's diagram
outputs.

\subsection{Scenario Exercise: A Real Engineering Issue}\label{sec:scenario-exercise}

The exercise centred on a previously-resolved defect chosen for matching
this thesis's target problem class: an implicit, event-driven
interaction invisible without reconstructing the call/event chain.

\textbf{Defect summary (generalised, anonymised).} A control service
exposes interfaces to a client via proxy objects created on connect. On
disconnect, most proxies are freed; one was not, due to a missing reset
in connection-handling code. After a later restart, the stale proxy
silently stopped delivering change-notification events. The defect was
visible only by tracing the event path from the reconnect handler,
through the stale proxy, to the downstream handler.

\textbf{Task.} Participants used the tool to answer three questions
about the already-fixed codebase: which files changed, which test
exercises the fix, and where the proxy objects are used. All three are
answerable from the static call/containment graph alone, making this a
check on the same static capability underlying \rqone\ and \rqtwo, now
on an unfamiliar, industrial-scale codebase. Participants correctly
identified the modified file, the relevant test, and the proxy's
construction/release sites --- the components underlying the actual fix.

\subsection{Interview Findings}\label{sec:interview-findings}

Four practitioners were interviewed: a software designer (P1), a product
owner for an embedded control service (P2), a senior architect for
inter-component synchronisation (P3), and an algorithm engineer (P4).
Interviews covered general usefulness and six diagram types: class
structure (class diagram, polymetric view), object interaction (call
graph, dependency diagram), and system overview (structural, sequence
diagram).

All four use AI coding assistants daily and saw value in
architecture/dependency information. P3 reported completing an
investigation in a five-minute session that had previously taken roughly
five months --- a striking self-report, but a single, unverified,
retrospective comparison, reported as anecdote rather than measured
effect. P2 noted the graph-based tool located a stub implementation
whose concrete definition lived outside the visible workspace, a failure
mode of code-search-based assistants. P4 found it most useful for
onboarding onto unfamiliar code and locating coverage gaps.

Feedback on diagrams converged on a pattern. Dependency and structural
diagrams were consistently rated useful, and in P3's case judged cleaner
than the team's existing modelling-tool output. Sequence diagrams were
valued for surfacing unexpected race conditions and cross-component
effects (P1, P3, P4), but all three found event/message labels (\eg a
generic \texttt{Subscribe}) too abstract without inline detail, and
requested source class/line number on hover. The class diagram and
polymetric view drew the most mixed feedback: three of four preferred
the plain class diagram, and the polymetric view's size-based encoding
was repeatedly flagged as measuring the wrong thing (P2, P4 suggested
cyclomatic complexity or requirements traceability instead). Call graphs
were the most polarising output --- redundant with existing build-system
views for P4, not what P3 expected, while P2 rated the equivalent
dependency diagram positively for indicating ``hops'' between
components.

Three requests recurred across interviews, independent of diagram type:
interactivity (static diagrams compared unfavourably to tools allowing
search/filter/collapse, P1--P3); non-code context such as timing,
deployment configuration and product variant (P2, P4) --- a direct
instance of the construct-validity limitation already noted for this
thesis's own evaluation (\cref{sec:h1-corrections}), since the graph
only contains what was extracted; and invocation friction, with some
participants needing unusually explicit phrasing to trigger the tool
(P1, P4).

\subsection{Relationship to This Thesis's Contribution}

The class diagram and polymetric view are contributed by other \ac{GDI}
members and fall outside this thesis's scope; they are reported only as
context. The findings most relevant here are the positive reception of
the dependency and structural diagrams (descended from the interface
dependency table, \cref{sec:idt}) and the actionable critique of
sequence-diagram labelling (relevant to \sysml generation,
\cref{sec:sysml-generation}). The hover-detail request is a low-effort
extension of the existing \texttt{sourceLineNumber} property already
carried on \texttt{CppCalls} edges (\cref{sec:utility}), noted as future
work (\cref{sec:future-work}).

The proxy-leak defect (\cref{sec:scenario-exercise}) is structurally
analogous to S4: an interaction architecturally present in source but
invisible at runtime unless a specific failure condition is exercised.
Participants resolved it using only the static graph --- the same
evidence source as C2, which recovers architecturally-present
interactions that traces are liable to miss
(\cref{sec:h1-c4c2}). This is informal, but convergent, evidence that
the H1 finding generalises beyond CPSCore.

The same architecture also motivates, without measuring, the C4$=$C2
prediction (\cref{sec:cpscore-scale,sec:h1-c4c2}). Unlike CPSCore, this
codebase uses proxy objects, virtual interfaces (118 events on one
representative interface) and dynamic connect/disconnect --- exactly the
cross-module dispatch scenario where C4$>$C2 would be predicted. P2's
stub-resolution observation is consistent with this mechanism, but the
session collected no dynamic evidence, so it cannot confirm C4$>$C2
here; that was measured only on CPSCore, via S4
(\cref{sec:h1-c4c3}). The industrial architecture motivates the
prediction; S4 tests it. Confirming C4$>$C2 on a system of this kind is
left to future work (\cref{sec:future-work}).

\section{Discussion: Cross-Cutting Pattern}\label{sec:discussion}

A single pattern seems to run across all four evidence sources: additional
evidence helps exactly when the primary source is missing something, and
is otherwise redundant.

\begin{description}
  \item[H1.] Static analysis already contains everything the trace
    provides on happy-path scenarios (C4$=$C2); combining becomes
    additive only where each source is blind to a different interaction
    (S4).
  \item[H2.] Guided instrumentation removes noise without losing
    coverage because the scenario scope correctly identifies what is
    needed; the panel confirmed this translates into perceived
    usability, with the one exception (S2 usefulness) occurring
    precisely where the removed information was itself useful for
    debugging.
  \item[H3.] The reconstructed diagram improves task performance only on
    the fault-path scenario where the trace is incomplete --- exactly
    where H1 shows combining adds value.
  \item[Practitioner session.] The proxy-leak defect was resolved using
    only the static graph, mirroring the mechanism H1 demonstrates on S4.
\end{description}

This suggests the pipeline's value is conditional on information gaps
rather than uniform: it earns its keep where a simpler approach has a
blind spot, which is a practically useful thing for engineers to know
when deciding whether the full pipeline is warranted.

The pattern also generates a testable prediction for future work: on a
codebase with cross-module virtual dispatch (unlike CPSCore), combining
should produce C4$>$C2 on \emph{all} scenario types, not just fault
paths, since the static graph would have blind spots even on happy-path
execution. The practitioner session's codebase (proxy-based, 118-event
interfaces, dynamic service binding) looks like exactly such a system,
and the engineers' difficulty resolving stub implementations behind
indirection is suggestive, if not conclusive, evidence of this blind
spot.

\section{Answering the Research Questions}\label{sec:rq-answers}

\subsection*{\rqone: Hybrid vs. Individual Sources}

\textit{How does combining static analysis and incomplete runtime traces
improve reconstruction accuracy and completeness?}

Combining improves reconstruction \emph{conditionally}, not universally.
On happy-path scenarios, static analysis alone matches the full
pipeline. On fault-path scenarios, combining improves over dynamic-only
analysis by filling in error-handling interactions a single trace is
unlikely to exercise. S4 shows the advantage is not limited to being
protective: once a boundary is introduced where each source is blind to
a different interaction, combining becomes strictly additive, beating
both sources at once (F1~$=$~1.000 vs.\ 0.667 for both C2 and C3). Here,
``combining'' denotes the set union of the two evidence sets computed at
evaluation time, not a merged graph. A thesis testing only happy-path
scenarios would have concluded combining adds no value at all, which is
why S4 was constructed specifically to isolate this precondition.

\subsection*{\rqtwo: Scenario-Scoped Querying}

\textit{How can scenario-scoped querying of the static call graph focus
reconstruction on only the behaviour relevant to a given scenario?}

Applying the scenario scope \emph{before} instrumentation, rather than
filtering afterward, reduces retained nodes by 50\% and edges by 70\%
while holding coverage at 1.000, demonstrating the mechanism works
structurally. The H3 results provide complementary evidence from the
consumer side: diagram-augmented context improves comprehension
specifically on the scenario where scoped querying recovers evidence
runtime tracing suppresses. The blinded expert rating confirms the
structural improvement also translates into perceived usability.

\section{Engineering Utility}\label{sec:utility}

Beyond precision/recall, a tool development thesis's primary claim is
that the tool is useful to engineers. Three utility properties were
observed during development.

\paragraph{Pipeline automation.} The full path from \Cpp\ source to a
\sysml sequence diagram runs by chaining eleven agent skills
(\cref{tab:skills}), each stateless and producing an auditable
intermediate artefact (\texttt{graph\_schema.txt}, a CSV, the final
\texttt{.sysml} file). An engineer can inspect the CSV row, find the
\texttt{sourceLineNumber} on the corresponding \texttt{CppCalls} edge,
and verify or override it directly without rerunning extraction. Skills
query the graph schema dynamically rather than hardcoding node labels,
so the same skill set works on any Renaissance-analysed codebase
unmodified.

\paragraph{Scale.} A single Cypher query extracted 2,574 ordered
interactions from CPSCore --- a task that would take an experienced
engineer several working days by hand. The same query identified the
cross-component call boundaries to instrument (nine caller functions
across five files); \texttt{csp\_matcher} then wrapped the corresponding
call sites (three find/replace rules plus a logging-macro patch) and
captured the trace, with no prior knowledge of the codebase structure
required.

\paragraph{Correctness preservation.} Inserting \texttt{csp\_matcher}
probes did not alter CPSCore's observable behaviour: all 1,425 test
assertions passed after instrumentation. The tool's reversibility
property (\cref{sec:csp-matcher}) allows instrumentation to be cleanly
removed, leaving the source tree byte-identical to its pre-instrumentation
state.

\section{Threats to Validity}

\subsection{Researcher Degrees of Freedom}\label{sec:threats-rdof}

H1 and H2 surfaced several points where a definitional choice could have
favoured a preferred result: ground-truth circularity, the timing of the
\texttt{SCENARIO\_PRIMARY} definition, F1 aggregation, scenario mix, and
the H2 noise/reference circularity
(\cref{sec:h1-corrections,sec:h2-corrections}). Each was caught before
being treated as final, which is reported here as a threat in its own
right. Catching these risks in this project does not guarantee an
equivalent study, run under time pressure or by a different team, would
catch them too.

\subsection{Internal Validity}

\begin{description}
  \item[Reference diagram subjectivity.] The reference diagrams reflect
    a single engineer's understanding per scenario; a different reviewer
    might choose different message granularity or lifeline selection.

  \item[Instrumentation coverage.] Instrumentation targets were selected
    by inspecting the static graph, not by coverage analysis. A
    different selection of entry points would likely produce a
    different trace and H1/H2 result.

  \item[Synthetic timestamps in the static baseline.] Timestamps in the
    C2 sequence table are synthetic (1\,ms spacing from an arbitrary
    base), since that table projects \texttt{CppCalls} edges rather than
    a run --- correct for ordering but not wall-clock behaviour. The
    per-scenario C3 traces, by contrast, carry real
    \texttt{csp\_matcher} wall-clock timestamps.
\end{description}

\subsection{External Validity}

\begin{description}
  \item[Single controlled system for H1/H2/H3.] CPSCore is a
    moderate-scale framework with a clean modular structure. The
    practitioner session provides encouraging qualitative evidence that
    the static-extraction approach transfers, but it is an unblinded,
    four-participant interview study, not a repeat of the controlled
    comparisons. Applying the full hypothesis tests to a second system
    remains the strongest way to strengthen external validity.

  \item[Renaissance dependency.] Static extraction depends on
    Renaissance, an internal TNO-ESI tool. Replicating the pipeline
    elsewhere would require an equivalent \Cpp\ semantic extractor
    emitting a compatible schema; the agent skills are schema-agnostic
    and would need only \texttt{get-neo4j-schema} re-run against it.

  \item[S4 is constructed, not found.] The strict-additivity result
    rests on a single scenario whose production code was written
    specifically for this experiment rather than discovered in
    CPSCore. It is reported and stratified separately for this reason.
    The finding is corroborated by an independent, naturally-occurring
    observation in the practitioner session, but a larger, more diverse
    sample of fault-path/dual-gap scenarios --- ideally from a codebase
    where such indirection is pervasive --- would give more confidence
    that the effect reflects the underlying mechanism rather than a
    property of S4 specifically.
\end{description}

\subsection{Construct Validity}

\begin{description}
  \item[Edge-level evaluation metric.] Precision and recall are measured
    over unique $\langle\text{sender},\,\text{receiver}\rangle$ pairs,
    abstracting over message count and temporal ordering. This may
    underestimate the practical importance of ordering accuracy in a
    sequence diagram; a metric accounting for sequence alignment is
    identified as future work.

  \item[Noise/coverage circularity in H2.] The noise metric shares a
    scope definition with the guided-instrumentation scenario scope
    (\cref{sec:h2-corrections}), inflating the apparent noise reduction
    where scope and reference were defined from the same pass. This is
    mitigated, but not eliminated, by S3's independently-defined
    falsification case.
\end{description}
