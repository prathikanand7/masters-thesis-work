## Ground Truth Calculation

**What it was on `main`:**
- Hand-authored `elements.json` per scenario, placed in `reference_diagrams/S1–S4/`
- Written by manually reading `.cpp`/`.h` files, then transcribing cross-component calls
- Scope was limited to edges where the **source** component matched the single "primary driver" (`SCENARIO_PRIMARY` filter) — e.g. only `Synchronization → X` edges counted for S1, even if `Aggregation → Logging` calls also fired
- No sequence-level GT existed; only the unordered interaction set was evaluated

**What it is now (on `feature/refactor`):**
- Automated pipeline in `experiments/scenarios/derive_gt.py`; produces two files per scenario:
  - `gt_interactions.json` — unordered set; union of static + dynamic edges, validated against the architectural diagram
  - `gt_sequence.json` — ordered list with all repetitions and an `evidence` field (`"dynamic"`, `"static"`, or `"both"`)
- Three evidence sources fused:
  1. **Renaissance Neo4j** — forward `CppCalls*` traversal from per-scenario entry-point functions (same Cypher as C2); finds what is statically reachable
  2. **trace_slice.txt** — runtime events captured during the actual test; only events whose `ClientFunction` is **reachable from the scenario entry points** are included (the `CYPHER_REACHABLE` query resolves this — it filters out test-fixture setup calls like `Aggregator.add` that fire before the scenario logic starts)
  3. **`architectural_diagrams/cpsCore_packages_annotated.puml`** — generated by `clang-uml` from the real C++ source; any edge whose `(src, tgt)` pair does not appear as an architectural `..>` arrow is rejected
- Static-only edges are **explicitly included** only when manually confirmed as exercised-but-suppressed at runtime (e.g. S4's `stream` call gated by `LogLevelScope(NONE)` — the `STATIC_ONLY_INCLUDE` dict captures this)
- The `primary driver` restriction is **removed** — any cross-component call that is (a) reachable from the entry points and (b) on a real architectural edge is GT

**Why this is better:**
- Manual GT is brittle, hard to reproduce, and biased — the author decides what "counts" subjectively
- The primary-driver filter was wrong in principle: if `Aggregation → Logging` fires during the scenario (S1a), it is a real interaction regardless of who initiated the top-level call; excluding it made GT artificially narrow and distorted F1 scores
- The new pipeline is reproducible: re-running derive_gt.py regenerates the same GT from the same Neo4j + trace data
- The reachability filter (excluding setup boilerplate) makes GT represent **scenario logic** rather than "anything the process does during a test" — this is a more scientifically meaningful ground truth
- The puml validation guards against spurious instrumentation events that cross non-architectural boundaries (no real boundary-crossing events have been rejected so far, but the check is there as a safeguard)

## Scenarios

### The clang-renaissance tool

- **`clang-exp`** is the custom instrumentation tool built for this project. It:
  1. Takes the Renaissance Neo4j call graph as input — specifically the **architectural interface map** (`cpsCore_packages_annotated.puml`, produced by `clang-uml` from the codebase) which lists which component-to-component call directions are architecturally valid
  2. Instruments only the **specific call sites** that cross the architectural boundaries defined in the map — it does NOT instrument all function calls; only calls at known cross-component interfaces
  3. Runs inside the test binary and emits pipe-delimited `[TRACE]` lines (format: `timestamp | eventName | clientComp | clientFn | serverComp | serverFn | relType | iface`) on every instrumented call that fires
  4. The output is formatted into the standard 8-column pipe-delimited format by format_trace.py

- **trace_slice.txt** (guided trace) = the output of `clang-exp` run against the scenario test — it contains only calls at architecturally-known cross-component interfaces; intra-component calls and non-architectural calls are never emitted
  - This is **pre-filtered by architecture** — the static graph was used *before* instrumentation to decide *which* call sites to probe
  - Result: very low noise; every event is a genuine cross-component interaction

- **full_trace.txt** = `clang-exp` run in **promiscuous mode** (instruments all detected cross-component call sites without pre-filtering against the architectural map) — captures more events including ones that were not known to be architectural in advance
  - Contains additional events that are valid cross-component calls but either: (a) test-setup boilerplate (e.g. `Aggregator.add`→`Logging` during fixture setup), (b) intra-module calls that span source files in the same component, or (c) calls on code paths not exercised by this specific scenario
  - Noisier but broader: used for the `C3f/C4f/C5f/C6f` comparison

### Scenario S1 → S1a + S1b split

- **On `main`:** A single scenario `S1` representing the `"Synchronized Runner Test"` happy path
- **On `feature/refactor`:** Split into two distinct scenarios to represent two qualitatively different test types:

  - **S1a** — `"Synchronized Runner Test"` (happy path): master drives two runners through `INIT → NORMAL → FINAL`; all succeed. Components: `Sync, Agg, Log`. Key interactions: `SynchronizedRunner→Aggregator.getAll`, `SynchronizedRunner→CPSLogger.flush`, `SynchronizedRunnerMaster→RAIILogStream.stream`. GT: 2 interactions (after excluding `Aggregator.add` setup noise), 8 sequence events
  - **S1b** — `"Synchronized Runner Timeout"` (failure path): one runner object exceeds the 1-second timeout; `CPSLOG_WARN` fires in `runStage`. The test calls `CPSLogger::instance()->setLogLevel(LogLevel::NONE)` which suppresses all log output at runtime — so C3 cannot see the `Sync→Log:stream` edge. C2 (static) finds it. This deliberately mirrors S4's dynamic-only gap: S1b has a **static-only-visible** interaction, while S4 has a **dynamic-only-visible** one (via `boost::signals2`). GT: 3 interactions, 5 sequence events

- **Why splitting is better:** A single S1 mixed happy-path and failure-path expectations into one undifferentiated scenario. The split allows separate evaluation of each test's interaction set, gives a symmetric counterpart to S4 (both involve a suppressed call, but in opposite directions: S1b static-only, S4 dynamic-only), and removes false precision in the aggregate F1 numbers

### S4 — Stage Event Bridge (GT revised)

**On `main`:**
- GT was hand-authored to include two interactions: `onStageEvent` (Sync→Agg via
  `boost::signals2`) and `stream` (Sync→Log via `CPSLOG_ERROR`)
- `[S4TRACE]` probes were manually added to `StageEventBridge.cpp` and
  `StageEventListener.cpp` to confirm edge coverage by tracing at source level

**On `feature/refactor`:**
- `[S4TRACE]` probes removed: they only confirmed edges already known from source
  reading, and manually instrumenting the code under study is coverage by hand —
  not a methodology result
- GT produced automatically by `derive_gt.py`: Neo4j was updated to include S4's
  call graph, then the same pipeline (static traversal + clang-exp trace + puml
  validation) ran as for every other scenario, yielding one interaction: `stream`
  (Sync→Log) confirmed by both static and dynamic

**Why better:**
- GT should only contain edges that the pipeline can in principle observe; an edge
  visible to neither static analysis nor runtime tracing is not a fair test case for
  either condition — it can only ever be a false negative for every condition, which
  inflates FN counts without distinguishing between methods
- Removing manually-added source probes ensures the trace files are produced entirely
  by the clang-exp pipeline, consistent with every other scenario

## Conditions (one by one)

### C1 — LLM-only

**On `main`:**
- Single LLM call; input was a hardcoded plain-English description of the scenario (no source code)
- Goal: "floor baseline — LLM reconstructs from world knowledge of C++ frameworks"
- Output: `elements.json` (interaction set only, no sequence)

**On `feature/refactor`:**
- Input changed from description to the **actual source snippet** (`source_snippet.txt` — the test file and relevant production files)
- Two chained LLM calls per scenario:
  - Call 1 (`PROMPT_INTERACTIONS`): identify the complete unique set of cross-component calls; rules explicitly cover RAII log macros, `CPSLogger.flush`, singleton exclusions, and failure paths
  - Call 2 (`PROMPT_SEQUENCE`): given the interactions from call 1, produce the full ordered sequence with repetitions (loop counts, thread counts, per-stage firings)
- Output: `interactions.json` (set) + `sequence.json` (ordered)
- Model deployment is read from `AZURE_OPENAI_CHAT_DEPLOYMENT_NAME` env var (was hardcoded)

**Why better:** A plain-English description is far weaker evidence than actual source code. The LLM cannot know specific method names, thread counts, or loop bounds from a verbal summary — it guesses and over-generates. With source code, C1 scores **1.000 on S1a** (perfect set and sequence); on `main` C1 scored 0.447 macro F1 overall. The two-prompt separation also prevents the LLM from conflating "which interactions exist" with "how many times does each fire" — a known prompt-engineering failure mode that produced wrong multiplicities in the combined prompt

### C2 — Static-only

**On `main`:**
- Primary data source: `sequence_dependency_table.csv` — a static export of the Renaissance call graph for the entire codebase; filtered by scenario component set
- Neo4j used only as fallback (and silently produced no results in practice)
- Global traversal: all `CppCalls` edges across the whole codebase were considered, not just those reachable from the test

**On `feature/refactor`:**
- CSV approach removed entirely
- **Test-scoped Neo4j traversal** using `SCENARIO_ENTRY` per scenario (e.g. `SynchronizedRunner.runSynchronized` for S1a): only `CppCalls*` edges reachable from the scenario's entry-point functions are returned
- Two Cypher queries: `CYPHER` (direct + one-hop via `CppImplements`) and `CYPHER_ORDERED` (variable-length path, depth 1–6, returns `min(length(path))` for ordering)
- `SCENARIO_EXCLUDE` dict filters known implementation-detail calls (`getLogLevel` in S4)
- `SCENARIO_ENTRY` includes indirect-dispatch seeds (e.g. `MultiThreadingScheduler.runSchedule` for S3) to compensate for `std::function`/thread opacity
- Primary-source filter removed (any cross-component edge in allowed set counts)
- Also outputs `sequence.json` with depth-ordered events (shorter call-graph path ≈ earlier execution)

**Why better:** Global traversal was the main source of C2 false positives — it found paths that exist in the codebase but are not exercised by the specific test. Test-scoped traversal reduces FPs significantly. On `main` C2 macro F1 was unreported cleanly, but locally C2 scores 0.738 macro — with remaining gaps from `boost::signals2`/`std::function` indirect dispatch that Neo4j cannot follow (a fundamental static analysis limitation, not a methodology flaw)

### C3 — Dynamic-only (guided trace)

**On `main`:**
- Filtered trace rows to only those where `src == SCENARIO_PRIMARY[scenario]` — only edges driven by the one primary component were counted
- Output: `elements.json` only

**On `feature/refactor`:**
- Primary-source filter **removed** — any cross-component pair in the allowed component set is included
- Both `interactions.json` (deduped set) and `sequence.json` (all trace events in order) produced

**Why better:** The primary-driver filter was methodologically unjustified — if `Aggregation → Logging` fires during the scenario, it is a real interaction regardless of whether Synchronization initiated the top-level call. The filter was silently hiding real GTs from evaluation. Removing it makes C3 more complete (recall improves) without adding noise (trace events on architectural edges are genuinely exercised calls). Locally C3 macro F1 = **0.865**; with sequence, C3 seq macro = **0.810**

### C3f — Dynamic-only, full trace (new, local only)

- Same as C3 but reads full_trace.txt instead of trace_slice.txt
- Contains more events including setup boilerplate and calls on non-scenario code paths
- C3f macro interactions = **0.831** vs C3 = 0.865 — guided is better
- C3f seq = **0.771** vs C3 = 0.810 — guided is better
- **Why this condition exists:** quantifies the value of guided (architecture-scoped) instrumentation vs unguided. The guided trace already had clang-exp pre-filter using the architectural map — C3f shows what you would get without that pre-filtering

### C4 — Static + Dynamic union

**On `main`:**
- C2 ∪ C3 union; used `elements_path` pointing to `elements.json`; no sequence

**On `feature/refactor`:**
- Same union logic but uses `interactions_path`; additionally copies C3's `sequence.json` as C4's sequence backbone (union adds C2 elements to the interaction set; ordering stays trace-derived)
- On `main` this was described as potentially beneficial for fault-path recall (the H1b/H1c hypothesis); locally it is now **explicitly shown to be harmful**

**Why it is worse (not better — this is the negative result):** C4 macro interactions = **0.764**, which is **lower than C3 alone (0.865)**. The union inherits all of C2's false positives — static over-approximation adds spurious edges — without the dynamic trace contributing any TPs that the trace didn't already have. On S4 specifically, C4 (0.500) is worse than both C2 (0.667) and C3 (0.667) because it accumulates both sources' FPs simultaneously. C4 and C3 score **identically on sequence** (0.810) because C4 just copies C3's trace sequence — the union adds nothing to ordering

### C4f — Static + Dynamic union, full trace (new, local only)

- C2 ∪ C3f; identical to C4 but with full trace as the dynamic source
- Interactions macro = **0.764** (same as C4, because C2's FPs dominate regardless of trace variant)
- Sequence macro = **0.771** (worse than C4 = 0.810 because C3f's noisier sequence backbone degrades ordering)

### C5 — Intersection C2 ∩ C3 (redefined from `main`)

**On `main`:** C5 was the "full LLM synthesis pipeline" — what is now C6 locally

**On `feature/refactor`:** C5 is **set intersection**: only interactions confirmed by **both** static (C2) and dynamic (C3) are retained; sequence = C3's trace events filtered to only intersection-confirmed interactions

**Why intersection is better than the old C5 / full pipeline:** The intersection exploits a key insight: if an edge is a false positive in C2, it almost never appears in C3's trace (FPs in static analysis are paths that don't execute); if an edge is a false positive in C3, it almost never appears in C2's graph (spurious trace events are filtered by puml validation). The intersection of two imperfect-but-complementary sources is cleaner than either alone. C5 macro interactions = **0.900** — the best of any condition for interactions. On S4 it scores **1.000**: `stream` (Syn→Log) is confirmed by both Neo4j and trace; `onStageEvent` (only in trace) and `subscribeStage` (only in graph) are eliminated. On S3 it drops to 0.500 because C2's indirect-dispatch gap means the `Util→Log:stream` edge is never in the static graph — intersection conflates "statically invisible" with "incorrect", which is a genuine limitation

### C5f — Intersection C2 ∩ C3f (new, local only)

- C2 ∩ C3f: same intersection but with full trace
- Interactions macro = **0.833** vs C5 = 0.900 — guided is better
- S4 drops from 1.000 to 0.667: the full trace adds `Agg→Log:stream` and `subscribeStage` noise events that also appear in C2's static graph, polluting the intersection with FPs that the guided trace would have excluded
- **Why this exists:** shows that guided instrumentation is not just helpful for C3 alone but also for the intersection — the clang-exp pre-filtering is doing genuine noise reduction that survives the set algebra

### C6 — Full pipeline, LLM synthesis (new, previously C5 on `main`)

**What it was on `main` (as C5):**
- Gave the agent C2 and C3 outputs with provenance tags; agent synthesised a merged diagram and produced both `elements.json` and `sequence.mmd`
- Reported macro F1 = 0.964 on the old scenario set (S1–S4 without S1a/S1b split and without the revised GT)

**On `feature/refactor` (as C6):**
- Same provenance-tagged C2+C3 input but **also includes the source snippet** (`source_snippet.txt`)
- Two-call chain: call 1 = interaction set (given C2 edges, C3 trace, source); call 2 = sequence (given interactions + C3 trace + source, LLM produces ordered sequence with setup events filtered out via source reasoning)
- Output: `interactions.json` + `sequence.json` + `sequence.mmd`
- Macro interactions = **0.871**; macro sequence = **0.913** (best condition for sequence)

**Why C6 is the best for sequence specifically:** The LLM uses the source snippet to identify that e.g. `Aggregator.add` calls in trace_slice.txt are test-fixture setup boilerplate (it can read the test body and see that those calls precede the scenario logic), and removes them from the sequence output. This is something C3, C4, and C5 cannot do mechanically — they have no semantic understanding of which caller is "setup" vs "scenario". C6's remaining failures on S4 and S3 are **pure LLM reasoning errors** (it over-generates FPs despite having all required evidence present in C2+C3+source); not evidence gaps. This is the saturation point for the current prompt design

### C6f — Full pipeline, full trace (new, local only)

- Same as C6 but reads full_trace.txt as the dynamic evidence
- Interactions macro = **0.900** (matches C5!) — the LLM partially compensates for the noisier trace by using the source to reason away setup events
- Sequence macro = **0.873** (lower than C6 guided = 0.913) — the noisier context degrades sequence more than interactions; the LLM can identify some setup interactions to exclude but cannot perfectly order a noisier event list
- **Note:** C6f results vary between LLM runs (non-deterministic); the reported values are from a single run

## Results and Demonstrated Hypotheses

### H1a — Intersection precision (confirmed, macro=0.900)

- **Claim:** C5 = C2 ∩ C3 achieves higher interaction F1 than either source alone (C2, C3) and than the LLM synthesis (C6), because each source's FPs are not shared with the other
- **Numbers:** C5=0.900 > C3=0.865 > C6=0.871 > C2=0.738
- **Why:** Each source has different failure modes — static analysis misses indirect dispatch, dynamic traces capture setup noise. Their overlap is precisely the true positives. On S4 this is exact: C2 has `subscribeStage` as FP, C3 has `Agg→Log:stream` as FP; neither FP appears in the other source, so intersection = GT exactly = 1.000

### H1b — Union is harmful (confirmed, C4 macro=0.764 < C3=0.865)

- **Claim:** C4 = C2 ∪ C3 does NOT improve over dynamic-only (C3) on interactions; static over-approximation introduces FPs that outweigh any recall gain
- **Numbers:** C4=0.764 < C3=0.865; on S4, C4=0.500 which is worse than both C2=0.667 and C3=0.667
- **Why:** C3 already captures all TPs that the trace can observe. C2 adds only its own over-approximate edges as FPs to the union. The union's recall gain is zero (C3 already has all exercised edges); its precision loss is real (C2's static FPs are now in the merged set). This directly contradicts `main`'s H1b/H1c claim that C4 achieves higher recall than C3 on fault-path scenarios

### H1c — LLM value is sequence-specific (confirmed, C6 seq macro=0.913)

- **Claim:** C6 is the best for sequence (0.913) but not for interactions (0.871 < C5=0.900). The LLM's unique contribution is filtering test-fixture setup events from the trace using source code reasoning
- **Why:** Setup events (calls from test body before scenario logic) appear in the trace at the same architectural call sites as scenario events. No mechanical filter can separate them without semantic understanding of the test code. The LLM reads the source snippet, recognises `Aggregator.add` as setup code, and omits those trace events from the sequence output. C5's intersection misses this: it scores 0.267 on S3 sequence because C2 doesn't have the indirect-dispatch edges, so intersection removes real events from the sequence along with the noise
- **C6's remaining shortcomings** are entirely reasoning errors (FPs on S4 where it infers extra calls not in GT, over-filtering on S3) — not evidence gaps. This proves C6 has saturated the available information; the next improvement is prompt engineering, not adding more data sources

### H1d — Guided instrumentation is the best noise reduction strategy (confirmed)

- **Claim:** Architecture-guided traces (C3, using trace_slice.txt from clang-exp with the architectural interface map) consistently outperform full traces (C3f) and post-hoc intersection of full traces (C5f). Using the static knowledge upfront at instrumentation time is more effective than using it as a post-hoc filter
- **Numbers:** C3=0.865 > C3f=0.831; C5=0.900 > C5f=0.833 on interactions; guided sequence always ≥ full for same condition; S4 is where the gap is largest (C5=1.000 vs C5f=0.667)
- **Why:** clang-exp instruments only architecturally-confirmed call sites, so the trace never contains non-architectural noise in the first place. Post-hoc filtering (C5f = intersection of full trace + static graph) can partially reproduce this but fails on S4 because the full trace contains events that happen to also be in C2's static graph but are setup boilerplate — the intersection keeps them as false TPs. Pre-instrumentation guidance is architecturally precise in a way that post-hoc filtering cannot fully replicate
