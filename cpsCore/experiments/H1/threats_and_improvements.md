# H1 Threats to Validity and Needed Improvements

---

## Threats to Validity

### Internal Validity

**T1 — Neo4j drives both the trace input and C5's scoring filter**

`trace_slice.txt` was generated by instrumenting sites identified through Neo4j's
CppCalls arch-edge query. C5's scoring filter is also derived from Neo4j
(`CppCalls*0..8` reachability). If Neo4j has a systematic blind spot at the
*instrumentation selection* stage (e.g. an architectural interaction that is
invisible to `CppCalls` static analysis), that interaction will be absent from
`trace_slice.txt` and therefore absent from both C5 and the GT simultaneously.
The evaluation cannot detect this class of omission.

*Partial mitigation*: We verified that manually instrumenting based on the excalidraw
arrows would produce exactly the same sites as `trace_slice.txt` — the two sets are
identical (documented in `architectural_diagrams/instrumentation_sites.txt`). This
establishes that Neo4j's arch-edge identification selects the same sites human architects
would choose, providing a design-level completeness check. It remains an empirical
observation rather than a formal guarantee.

**T2 — GT and C5 share the same evidence source**

Both the GT and C5 derive their interaction set from `trace_slice.txt`. C5's perfect
score demonstrates that Neo4j `CppCalls*0..8` reachability is an accurate scope
filter for Neo4j-instrumented trace events. It does not demonstrate that C5 can
discover architectural interactions independently of the Neo4j instrumentation
strategy. The evaluation measures *filter quality given a fixed trace*, not
*autonomous interaction discovery*.

*Partial mitigation*: C5 and the GT use entirely independent scoping mechanisms —
Neo4j `CppCalls*0..8` reachability (automated static-graph traversal) vs
human-drawn `SCENARIO_ENTRY` functions — that agree completely. This cross-validates
the filter's correctness from two independent sources of architectural knowledge,
even though both operate over the same `trace_slice.txt` event pool.

### External Validity

**T3 — Single small-scale codebase**

All four scenarios come from one framework (cpsCore, 6 components). Each scenario
contains 2–3 interaction types. Macro F1 averages are dominated by single edges.
No statistical inference is possible at this scale.

**T4 — Unit tests, not operational scenarios**

The scenarios are GTest unit tests, not real system execution paths. Production
scenarios may involve more components, longer call chains, dynamic loading, or
concurrent dispatch patterns not represented here.

**T5 — cpsCore-specific failure modes**

The characterised failure modes (Neo4j blind to `std::function` dispatch, LLM
missing `MultiThreadingScheduler.cpp`) are specific to cpsCore's design patterns.
Whether these failure modes generalise — and in what proportion — is untested.

---

### Construct Validity

**T6 — Information asymmetry between conditions**

The four conditions access fundamentally different amounts of architectural
information:

| Condition | Information available |
|---|---|
| C1 | Source snippet only (hand-selected subset of files) |
| C2 | Full Neo4j static call graph of the codebase |
| C3 | Complete runtime execution trace |
| C4 | Neo4j-targeted instrumentation (`trace_slice.txt`), no reachability filter |
| C5 | Neo4j-targeted instrumentation + Neo4j reachability filter |

This is not a controlled comparison of *methods* applied to the same information —
it is a comparison of *information regimes*. The observed ordering (C5 > C4 > C3 > C2 > C1)
reflects information advantage as much as method quality.

**T7 — C2's entry-point ordering is a weak proxy for execution order**

C2 sequences calls in SCENARIO_ENTRY traversal order — each entry point's cross-component
callees are appended in the order they appear in the Neo4j query result. This bears
no principled relationship to actual runtime execution order. A DFS over the full
static call graph was attempted but is non-trivial because the Neo4j schema separates
function declarations from definitions (`CppCalls` edges target declarations;
`CppImplements` edges link declarations to definitions), making multi-hop traversal
require alternating edge types that cannot be expressed as a simple variable-length
Cypher path. C2's sequence scores (macro seq 0.536) still reflect this limitation
more than a fundamental ceiling for static analysis.

## Needed Improvements

**I1 — Multiple codebases**
Replicate on at least 3–4 independent systems with different architectural styles
(event-driven, layered, microservices) to test generalisability of the failure
modes and the C5 cross-validation result.

**I2 — Larger scenarios**
Design scenarios with 10+ architectural interactions so that macro averages are
statistically meaningful and conditions are tested under higher complexity.

**I3 — Blind GT derivation**
GT should be produced by someone independent of the condition designers to prevent
unconscious alignment of GT scope assumptions with condition capabilities.

**I4 — Independent GT trace**
Derive the GT from a separately collected full execution trace filtered by an
independent oracle (e.g. manual inspection, coverage-guided instrumentation)
rather than from `trace_slice.txt`. This would allow the evaluation to detect
whether C5 can recover interactions it did not select at instrumentation time.

**I5 — Improved C2 ordering** *(partially addressed)*
Current implementation uses SCENARIO_ENTRY order (entry-point traversal order), which
improved macro seq from 0.439 to 0.536. A full call-tree DFS was attempted but blocked
by the CppCalls/CppImplements declaration-definition split in the Neo4j schema.
A principled next step would be a two-step Cypher expansion that explicitly traverses
`CppCalls → (decl) ← CppImplements → (def)` pairs to reconstruct the call tree,
enabling DFS pre-order sequencing.

**I6 — Quantify Neo4j blind spot coverage**
Enumerate all indirect-dispatch edges in cpsCore that Neo4j's `CppCalls` graph
misses (via `std::function`, `boost::signals2`, thread pools, etc.). This bounds
T1: knowing how many architectural interactions are systematically invisible to
both C5 and the GT narrows the scope of the undetectable omission.

**I7 — Sensitivity analysis for instrumentation completeness**
Deliberately omit some instrumentation sites from `trace_slice.txt` and measure
C5 degradation, to characterise how sensitive C5's perfect score is to the
completeness of the underlying Neo4j arch-edge identification.

**I8 — C1 snippet quality** *(partially addressed, then reverted)*
Engineered snippets for S1b, S2, and S3 were written to include complete execution paths
(addOptional chain, ObjectHandleContainer aggregation callbacks, SynchronizedRunner
runSynchronized body) and achieved C1=1.000/1.000 on all four scenarios. These were
subsequently reverted to natural snippets after recognising that engineered snippets
function as oracle hints — they pre-answer the question the LLM is supposed to reason
about, making the score scientifically invalid.
Current state (natural snippets): macro intr=0.964, seq=0.935. S3 scores 0.857/0.740
due to one FP: `stop()` is shown in the snippet with a `CPSLogger::flush` call, which
the LLM correctly reads as an interaction, but `stop()` is invoked by test teardown
and excluded from the GT by the reachability filter. Fixing this without engineering
the snippet requires either excluding `stop()` from the snippet (changes scope) or
accepting the FP as a genuine limitation of source-only analysis.

## Critical Observations

The following observations question the scientific novelty of the H1 findings and
should be considered before presenting or extending this work.

**Most findings confirm expected behaviour, not novel results.**

- *C5 perfect scores are near-tautological.* Instrumenting the exact right call sites
  and filtering to those same sites yields perfect precision and recall almost by
  definition. The score validates the mechanism works correctly, not that it discovered
  anything independently.

- *C6 = C5 is unsurprising.* A larger event set filtered by the same accurate scope
  filter naturally produces the same result. This confirms filter correctness, not a new
  insight about instrumentation strategy.

- *C3 vs C4 on interactions is obvious.* Adding guided instrumentation without a filter
  removes intra-component noise but leaves setup noise intact.

- *C2 missing `std::function` dispatch is a known C++ static-analysis limitation.*
  This is documented in the Renaissance literature and is not specific to this codebase
  or this experiment.

- *C1 vs C3 comparison is not a controlled experiment.* The two conditions operate on
  fundamentally different information (static source snippet vs full runtime trace).
  The 0.964 vs 0.914 comparison reflects information availability as much as method quality,
  and C1 is a single non-deterministic LLM run with no statistical grounding.

**The two findings with genuine scientific value:**

1. **The bijection between Neo4j arch-edge query results and human excalidraw
   instrumentation choices** (documented in `architectural_diagrams/instrumentation_sites.txt`)
   is an empirical validation that Neo4j's automated static analysis identifies the same
   architecturally significant call sites a human expert would select. This is not obvious —
   it depends on the quality of the Renaissance schema and the cpsCore coding conventions —
   and it is a prerequisite for trusting C5's instrumentation completeness.

2. **The Neo4j reachability filter (`CppCalls*0..8`) acts as an automated scope
   discriminator** that separates test-setup boilerplate from scenario logic without
   manual annotation. Its agreement with the independently derived `SCENARIO_ENTRY`
   filter (drawn from excalidraw diagrams by a human) is the cross-validation result
   that gives C5's perfect scores their meaning. Whether this generalises to other
   codebases or test frameworks is the open question that I1–I4 are designed to address.
