# H3 — Reconstructed Models Improve Agent Task Performance

**Hypothesis:** Agents equipped with reconstructed structural and sequence diagrams
perform software engineering tasks more accurately and efficiently than agents
relying only on source code and execution traces.

**Task 1 only (scope):** Scenario comprehension — components identified, interactions
identified, completeness of explanation.

**Design:** a single holistic experiment with **five conditions**, isolating source
code, execution traces, and the reconstructed C5 diagram both individually and in
combination, run across the same four scenarios (S1, S2, S3, S4).

## Scenarios

| ID | Name | Primary component | Path type | Components | Interactions |
|----|------|-------------------|-----------|:----------:|:------------:|
| S1 | Aggregator notification chain | Synchronization | happy | 3 | 4 |
| S2 | Configuration property mapping | Configuration | happy | 2 | 1 |
| S3 | Multi-component runner orchestration | Synchronization | happy | 4 | 3 |
| S4 | Stage event pub-sub bridge (constructed) | Synchronization | dual-gap | 3 | 2 |

S4 is a constructed scenario (new production code, not found-in-the-wild — see H1's
S4 writeup) where static analysis and runtime tracing are each blind to a *different*
interaction. Its C5 element list is generated by the **provenance-aware** C5
pipeline (see `experiments/H1/README.md`, "C5 Methodology"), which retains **both**
reference interactions (`onStageEvent` and `stream`). S4 is therefore a case where
source alone and traces alone can each miss something the diagram does not.

## Conditions

| Condition | Agent receives | Response folder |
|-----------|----------------|------------------|
| Source only | relevant source snippets | `responses/C_source_only/` |
| Trace only | runtime trace slice | `responses/Aprime_traces_only/` |
| Diagram only | C5 interaction list | `responses/D_diagram_only/` |
| Source + Trace | source snippets + runtime trace slice | `responses/A_no_diagrams/` |
| Source + Trace + Diagram | source snippets + runtime trace slice + C5 interaction list | `responses/B_with_diagrams/` |

**C5 context format**: `elements.json` (output of H1 C5 full-pipeline) formatted as
`Source -> Target : Interaction` lines. **NOT** `.sysml` files — C5 only produces
`elements.json`.

Three of the five conditions (trace only, source + trace, source + trace + diagram)
reuse real Azure OpenAI responses collected during earlier isolated A/B runs against
this same harness and ground truth; the two remaining conditions (source only,
diagram only) were newly collected for this holistic design so that all five points
in the design space — source alone, traces alone, diagram alone, and both pairwise
combinations that lead up to the full-context condition — are populated with real,
non-fabricated data.

## Discussion

**Diagram-only is the standout result of this experiment for what it does
*without* any other context.** With *no* access to source code and *no*
access to the runtime trace — only the C5 interaction list — the agent
achieves perfect component recall (12/12), perfect interaction recall
(10/10), and perfect completeness (5.00/5.00) on all four scenarios,
including S4, the constructed dual-gap scenario. This is a genuinely
strong result in favor of H3: a reconstructed diagram alone is not merely
a helpful supplement, it can be a *sufficient* — and, on its own, a
highly reliable — basis for scenario comprehension.

**Source + trace + diagram matches diagram-only exactly, on every
metric.** Both of the two conditions that include the C5 diagram achieve
perfect component recall, perfect interaction recall, and perfect
completeness (5.00/5.00) on all four scenarios — an earlier pass had
rated source + trace + diagram's S1 completeness at 4/5, but re-reading
that response against the trace slice it was given shows every named
function (`AggregatableRunner.notifyAggregationOnUpdate`,
`SimpleRunner.runAllStages`/`runStage`/`runStages`,
`SynchronizedRunner.runSynchronized`, `SynchronizedRunnerMaster`'s
constructor/`runAllStages`/`runStage`) is a verbatim, accurate match to
`S1/trace_slice.txt` — there was no rubric-justified basis for scoring it
below 5, so the value was corrected (see "Scoring notes"). Adding source
and trace on top of the diagram therefore neither helps nor hurts on this
scenario set: the diagram alone already saturates every metric, so there
is no remaining headroom for the additional context to improve on.

**Source-only has the weakest interaction recall of all five conditions (0.800).**
The agent's narrative under-reports macro-based/log-call interactions
(`RAIILogStream.stream`) on S1 and S4 even though the source snippets provided
contain explicit comments identifying them. This is not a lack of access — the
information is present in the prompt — but a tendency for source-grounded
narration to foreground the "main" control-flow path (the trace-confirmed,
listener-connected path) and omit secondary, log-only branches unless the model is
specifically prompted to enumerate every call site. Trace-only avoids this failure
mode on S1 (it directly observes the log emission event when it fires) but repeats
it on S4, where the trace itself never captures the suppressed edge (`LogLevel::NONE`).
Diagram-only avoids it everywhere, because C5 surfaces the interaction as an
explicit, labeled edge regardless of whether it was exercised at runtime or requires
source-level inference to notice.

**Source + trace and source + trace + diagram reproduce the same pattern as the
single-source conditions.** Source + trace scores identically to trace-only in
aggregate (0.917/0.900) — adding source snippets to the trace does not fix the
missed `stream` edge on S4, for the same source-narration reason described above.
This is a more specific failure than simply "having both sources does not
help": reading the source+trace response directly (`A_no_diagrams/S4_response.txt`)
shows the agent implicitly treats the trace as the more authoritative source
whenever the two disagree on what happened, rather than reconciling source
and trace as two independent, equally weighted evidence streams. On S4, the
source snippet is annotated with an explicit comment identifying the
`CPSLOG_ERROR` call as "source-visible (static analysis sees this call
unconditionally)" and separately noting it is "suppressed at runtime" — but
the response narrates only the trace-confirmed, listener-connected path and
never surfaces the suppressed branch as a distinct interaction at all. Having
source access in addition to the trace does not correct this: once the
trace supplies a concrete, single "here is what happened" account, the agent
appears to anchor on it and downweight the source-only branch as background
detail rather than an independent fact to reconcile against the trace.
Source + trace + diagram recovers to perfect recall (1.000/1.000). The
diagram avoids the same failure mode not by *outweighing* the trace with a
competing narrative, but by removing the asymmetry that produces it in the
first place: C5's interaction list presents every edge as a flat, equally
weighted fact (`Source -> Target : Interaction`) with no "this is what
actually happened at runtime" framing for the model to treat as more
authoritative than anything else. There is consequently nothing for the
agent to implicitly rank above the suppressed edge, and no comment-driven
false assumption for it to build a hallucinated priority ordering on top
of. This confirms that
it is specifically the **diagram** — not source, not traces, not their combination
— that reliably supplies the missing edge across every condition it appears in,
and suggests the mechanism is a de-biasing one: the diagram guides the model
with flat, unranked facts rather than implicitly signalling that one evidence
source (the trace) should be trusted over another (the source) when the two
are read together.

**Completeness mirrors the recall pattern exactly.** The two conditions that
include the diagram (diagram-only and source + trace + diagram) both achieve
perfect completeness (5.00) on every scenario, including S4. Source-only,
trace-only, and source + trace are tied at 4.25, all three losing a point
specifically on S4 — the same scenario where they also lose interaction
recall, for the same underlying reason (the missed `stream` edge).

## Conclusion

**H3 is supported, with the strongest and most direct evidence coming from the
two conditions that include the C5 diagram.** Across all four scenarios —
happy-path (S1–S3) and the constructed dual-gap scenario (S4) — diagram-only
and source+trace+diagram achieve identical, perfect scores on every metric
(component recall, interaction recall, and completeness), including
combinations that additionally include source code and/or execution traces.
These are the only two conditions that achieve perfect recall on every
scenario; the three conditions without the diagram (source-only, trace-only,
source+trace) are all held to 0.917/0.800–0.900 recall by the same recurring
failure — under-reporting the log-suppression interaction that the diagram
states explicitly. This is a more precise and more strongly supported
conclusion than the fragmented "S4-only" evidence from the earlier
two-experiment design: here, the diagram's benefit is visible not only on S4
but as the distinguishing factor across the *entire* condition-by-condition
comparison, and diagram-only's standalone perfect performance shows the
reconstructed model is not merely a supplement to source and traces but can
substitute for both.

## Scoring notes

- **Component matching**: case-insensitive substring (e.g. "Logging" in response)
- **Interaction matching**: source + target + **bare method name** (part after `.`) must
  all appear in the response; e.g. `"Aggregator.getAll"` -> check for `"getAll"`.
  This matches the H1 bare-name scoring convention.
- **Completeness (1-5)**: rated using `scoring/rubric.md`. The trace-only and
  source+trace values are carried over unchanged from the original blind
  rating pass (blinded files in `scoring/blind_responses/`, decode key in
  `scoring/decode_key.md`). The new source-only and diagram-only values were
  rated in a single, non-blind pass by the same rater applying the same
  rubric — disclosed explicitly as a methodological difference; a second,
  blind rater re-scoring these two conditions would strengthen the result.
- **Correction (source+trace+diagram, S1):** originally rated 4/5 in the same
  non-blind pass. On direct challenge, the response was re-read against
  `S1/trace_slice.txt` and found to cite every named client function
  (`AggregatableRunner.notifyAggregationOnUpdate`, `SimpleRunner.runAllStages`/`runStage`/`runStages`,
  `SynchronizedRunner.runSynchronized`, `SynchronizedRunnerMaster`'s
  constructor/`runAllStages`/`runStage`) correctly and completely, with no
  errors or omissions — a textbook 5 by the rubric's own criteria. There was
  no rubric-justified reason for the lower score, so it was corrected to 5
  and all dependent tables/aggregates updated. Disclosed here rather than
  silently edited, consistent with this project's practice elsewhere
  (H1/H2 circularity and artifact disclosures).
- **S4 scoring caveat:** S4's completeness scores (all conditions) were assigned
  in a single, non-blind pass when S4 was added, not through the blind
  procedure used for S1–S3.

## Planned addition: full-trace trace-only condition (needs XRay traces)

All current conditions that include a trace use `trace_slice.txt` (guided,
scenario-scoped). The "trace only" condition (A') therefore receives a clean,
noise-free trace. The planned addition is:

**A'' — Full trace only:** give the agent the per-test full trace (all events that
fire during that specific test, not just the guided slice) and score normally.

This answers: *does the agent's comprehension degrade when given a noisy full trace,
or can it filter the irrelevant calls itself?*

- If A'' scores the same as A': the agent absorbs the noise and guided instrumentation
  only matters for human readers (H2), not for the LLM pipeline.
- If A'' scores lower: guided instrumentation helps both humans (H2) and the LLM
  downstream task (H3), making the full chain — instrumentation scope → trace quality
  → agent performance — measurable end-to-end.

Requires: per-test full traces from LLVM XRay (see H2 README, "Future work" section).
No harness changes needed beyond swapping `trace_slice.txt` for the full trace file.
