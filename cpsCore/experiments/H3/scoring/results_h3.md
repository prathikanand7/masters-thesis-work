# H3 Evaluation Results ‚Äî LLM Scenario Explanation

Generated by `experiments/H3/scoring/evaluate_all_h3.py`.

---

## Research Question

**H3**: Does providing the LLM with a static architecture diagram (produced by the H1
pipeline) improve its ability to explain which components interact in a given test scenario
and why?

---

## Ground Truth Methodology

H3 ground truth is derived from the H1 `reference_sequence.json` files (trace-confirmed,
architecturally validated interactions). For each scenario the GT contains:

- **components** ‚Äî the set of cpsCore packages involved in the test
- **interactions** ‚Äî directed cross-component calls `(source, target, interaction)`
- **reference_explanation** ‚Äî a human-written explanation used during derivation

Scoring is **recall-only** (the LLM produces free-text; no FP penalty):
- **Component recall** ‚Äî fraction of GT components mentioned anywhere in the response
- **Interaction recall** ‚Äî fraction of GT interactions whose `(source, target, bare_method)`
  triple can be found in the response (all three terms must appear)
- **Completeness** ‚Äî holistic 1‚Äì5 manual rating (not automated)

### Per-scenario ground truth

| Scenario | Test | Components | GT interactions |
|----------|------|-----------|----------------|
| S1a | `"Synchronized Runner Test"` | Synchronization, Aggregation, Logging | Agg‚ÜíLog:stream, Sync‚ÜíAgg:getAll, Sync‚ÜíLog:flush |
| S2 | `"Optional test"` | Configuration, Logging | Config‚ÜíLog:stream |
| S3 | `"MultiThreaded Test 1"` | Synchronization, Aggregation, Utilities, Logging | Agg‚ÜíLog:stream, Sync‚ÜíAgg:getAll, Util‚ÜíLog:stream |
| S4 | `"Stage Event Bridge Pub-Sub Routing"` | Synchronization, Aggregation, Logging | Sync‚ÜíLog:stream, Sync‚ÜíAgg:onStageEvent |

> S4 is a deliberately constructed dual-gap scenario: `stream` is only visible statically
> (suppressed at runtime by `LogLevelScope(NONE)`), while `onStageEvent` is only visible at
> runtime (boost::signals2 dispatch, no static call site). It requires both evidence sources
> ‚Äî or the architectural diagram ‚Äî to achieve full recall.

## Key Findings

1. **Condition B (with C5 diagram) achieves perfect macro recall** on both components
   (1.000) and interactions (1.000) across all four scenarios.

2. **Condition A performs equally well on S1a‚ÄìS3** ‚Äî the LLM can correctly identify
   components and interactions from source + trace alone when the test scenario is
   straightforward.

3. **S4 is the critical discriminating scenario**: Condition A achieves only 0.667 component
   recall (misses `Aggregation`) and 0.500 interaction recall (misses `onStageEvent`).
   - Without the diagram the LLM cannot infer the boost::signals2 dispatch from
     `publishStage` ‚Üí `StageEventListener.onStageEvent`.
   - Condition B, given the C5 diagram that shows this edge, correctly includes it.

4. **The diagram adds value precisely where evidence is incomplete** ‚Äî when at least one
   interaction is invisible in both source and trace (S4), the architectural diagram
   bridges the gap. When all interactions are directly observable (S1a, S2, S3), the
   diagram provides no additional benefit.

5. **H3 hypothesis is supported for hard scenarios**: providing the C5 static+dynamic
   architecture diagram improves LLM explanation quality from partial to complete recall
   in the dual-gap (S4) scenario.
